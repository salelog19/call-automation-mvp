# MVP колтрекинга

## 1. Формулировка MVP

Показывать уникальный номер на сайте в зависимости от источника трафика и сохранять связку:

`visit -> source -> shown_number -> call`

## 2. Какие данные должны сохраняться

Для каждой сессии/визита:

- `ym_uid`
- дата и время визита
- `utm_source`
- `utm_medium`
- `utm_campaign`
- `utm_term`
- `utm_content`
- `referrer`
- URL страницы входа
- `session_id` или другой идентификатор визита
- показанный подменный номер телефона

Для каждого звонка:

- дата и время звонка
- подменный номер, на который позвонили
- номер телефона клиента
- идентификатор звонка у провайдера телефонии
- статус звонка
- длительность звонка, если провайдер её передаёт

## 3. Основной пользовательский сценарий

1. Посетитель переходит на сайт по рекламе или из другого канала.
2. На сайте определяется источник трафика по UTM-меткам и referrer.
3. Сниппет на сайте получает `ym_uid`.
4. Сниппет обращается к backend-сервису за номером.
5. Backend назначает свободный номер из пула.
6. Backend сохраняет визит в базу.
7. Сайт подменяет номер телефона на странице.
8. Пользователь звонит на показанный номер.
9. Провайдер телефонии отправляет webhook о звонке.
10. Система сохраняет звонок и связывает его с визитом через подменный номер и временное окно.

## 4. Что входит в MVP

- 1 проект / 1 сайт
- пул из нескольких подменных номеров
- JS-сниппет для подмены номера
- backend endpoint для выдачи номера
- сохранение визитов в `Supabase`
- webhook-приём звонков
- сохранение звонков в `Supabase`
- базовая атрибуция звонка к визиту

## 5. Что не входит в MVP

- мультиарендность с кабинетами клиентов
- сложная ролевая модель
- биллинг
- расширенная аналитика
- распределение номеров между множеством сайтов
- полноценная CRM
- сложная админка

## 6. Логика атрибуции в MVP

Базовое правило:

- при заходе на сайт пользователю выдаётся подменный номер
- этот номер закрепляется за визитом на ограниченное время
- при входящем звонке система ищет последний активный визит, связанный с этим номером

Предварительное окно закрепления номера:

- `30 минут`

Это значение можно изменить после первых тестов.

## 7. Минимальные компоненты системы

### Сайт

- вставленный JS-сниппет
- DOM-подмена телефонного номера

### Backend

- `POST /assign-number`
- `POST /call-webhook`
- логика выбора номера
- логика сохранения визита
- логика привязки звонка к визиту

### Supabase

- таблицы для проектов, номеров, визитов и звонков

### n8n

- опционально принимает webhook от телефонии
- пересылает/обрабатывает событие
- отправляет уведомления в Telegram или CRM

## 8. Критерии готовности MVP

MVP считается рабочим, если:

1. При заходе на сайт номер на странице меняется.
2. В базу сохраняются `ym_uid`, UTM-метки, дата/время визита и показанный номер.
3. При звонке в базу сохраняются дата/время звонка, номер клиента и номер, на который звонили.
4. Можно связать звонок с визитом и понять источник лида.

## 9. Первый технический шаг

Сначала реализовать только цепочку визита:

1. получить `ym_uid`
2. считать UTM-метки
3. запросить подменный номер
4. сохранить визит
5. заменить номер на сайте

После этого подключать телефонию и webhook звонков.
