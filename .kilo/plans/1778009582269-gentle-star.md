# Plan: GET /api/visits endpoint

## Architecture Analysis
- Extend existing Fastify backend with a new read-only endpoint.
- Follow current project structure: separate routes, services, and TypeScript models.
- Use parameterized SQL queries via pg Pool (already initialized in db.ts).
- Return JSON array with pagination metadata.
- No changes to database schema required; use existing `visits` table.
- Keep backend-first approach: implement API before any UI consumption.

## Minimal MVP Scope
- Provide a list of visits with basic filtering and pagination.
- Do not include complex analytics, joins, or aggregations in v1.
- Focus on stability and correctness; avoid over-engineering.
- Initial implementation will serve as foundation for admin panel and analytics UI.

## Response Shape
JSON object with:
```ts
{
  visits: Array<{
    id: string;
    projectId: string;
    sessionId: string;
    ymUid: string | null;
    utmSource: string | null;
    utmMedium: string | null;
    utmCampaign: string | null;
    utmTerm: string | null;
    utmContent: string | null;
    landingUrl: string | null;
    referrer: string | null;
    shownPhoneNumber: string;
    visitedAt: string; // ISO timestamp
    assignmentExpiresAt: string; // ISO timestamp
    createdAt: string; // ISO timestamp
  }>;
  pagination: {
    total: number;
    limit: number;
    offset: number;
  };
}
```
- Fields are selected directly from `visits` table.
- Timestamps are serialized to ISO strings (consistent with existing API).
- Nullable fields reflect possible absence of UTM or Yandex.Metrica data.

## Pagination
- Query parameters: `limit` (default 50, max 100) and `offset` (default 0).
- Alternative: `page` and `pageSize` if team prefers; but limit/offset is simpler and matches current project style.
- Response includes `total` count of filtered records for client-side pagination calculation.
- Enforce maximum limit to prevent oversized queries.

## Filters
Supported query parameters (all optional):
- `projectId`: UUID to filter visits for a specific project.
- `dateFrom`: ISO date string (inclusive) for `visitedAt >=`.
- `dateTo`: ISO date string (inclusive) for `visitedAt <=`.
- `utmSource`: exact match on `utm_source`.
- `utmMedium`: exact match on `utm_medium`.
- `utmCampaign`: exact match on `utm_campaign`.
- Filters are combined with AND logic.
- No OR or complex operators in MVP.

## Tables Used
- Primary: `visits` (contains all required fields for MVP).
- No joins needed; `shown_phone_number` is already denormalized in visits.
- Future expansions may join `tracking_numbers` or `projects` for enrichment.

## Files Needed
1. `src/routes/visits.ts` – Fastify route registration, handler, validation.
2. `src/services/get-visits.ts` (optional) – encapsulate SQL query and pagination logic.
3. Update `src/app.ts` to register the new route (e.g., `app.register(registerVisitsRoute, { prefix: '/api' })`).
4. Add TypeScript types if not existing (e.g., `VisitListItem`, `VisitsResponse`) in a shared types file or within the route file.

## Potential Risks
- **Performance**: Missing indexes on filtered columns (`projectId`, `visitedAt`, utm columns). Mitigation: ensure indexes exist; add if missing as a separate low-risk task.
- **Security**: SQL injection – mitigated by using parameterized queries via pg.
- **Breaking Changes**: Adding a new endpoint is additive; no existing contracts affected.
- **Data Exposure**: Ensure no sensitive fields (e.g., session tokens) are leaked; visited fields are already safe for analytics.
- **Versioning**: Consider future API versioning; but for MVP, keeping under `/api` is acceptable.
- **Deploy Safety**: No changes to Dockerfile, package.json, or env variables; only additive code.

## How to Check Endpoint
1. **Compile**: `npm run build` succeeds without new TypeScript errors.
2. **Deploy**: Deploy to Coolify (or local) and verify service starts.
3. **Manual Test**:
   - `GET /api/visits` returns 200 with empty array if no data.
   - Insert a test visit via existing assign-number flow, then fetch list and verify item appears.
   - Test each filter individually and in combination; confirm SQL correctly filters.
   - Test pagination: limit/offset returns expected slices; total matches filtered count.
   - Test invalid parameters (e.g., non‑UUID projectId, invalid date) return 400 with validation error.
   - Ensure existing endpoints (`/health`, `/assign-number`, etc.) remain unchanged.
4. **Automated**: Add a basic test in test suite (if exists) or plan for future test coverage.
5. **Logs**: Verify structured logs show request handling (if logging middleware enabled).

## Compliance with Project Rules
- Minimal changes: only additive files and one registration line in app.ts.
- No unrelated files touched.
- Follows existing style: TypeScript, Fastify decorators, parameterized SQL.
- Backend‑first: API ready before any UI.
- Deploy‑safe: no modifications to Docker, dependencies, or env.
- Avoids overengineering: no complex DTOs, no ORM, no caching layer in MVP.
